# CASE-03: Shipped Mock Data After 11 Red Team Rounds

## 1. The Situation

A project (kz-engage) had completed a full COC development cycle. It passed 11 red team rounds. It passed 94 tests. It passed 2 consecutive clean convergence rounds -- the criterion for declaring red team complete. By every metric the COC process measured, the project was ready to ship.

Yet when a human reviewed the deployed frontend, pages were displaying random numbers generated by `generateHourlyOccupancy()` instead of data from the real API. Architecture plans for a data fabric and ML fabric had been approved during analysis but never implemented. Todos had been marked complete because the relevant files existed on disk, not because the features were wired end-to-end.

## 2. The Trigger

The trigger was a post-mortem examination that asked a specific question: how did a project pass every automated and semi-automated quality gate while shipping with fake data? The answer was not a single failure but five compounding process failures, each of which was individually reasonable but collectively catastrophic.

The attentional pattern is **coverage-versus-quality blindness**: all existing validation checked whether the code that existed was good, but none checked whether the code that should exist actually did.

## 3. The Move

The move is **spec-coverage-audit** combined with **frontend-mock-detection** and **evidence-based-completion**. The post-mortem identified five specific process gaps and closed each one with a targeted artifact change:

1. Red team checked code quality but not spec coverage. Fix: added a "Spec coverage audit" as step 1 of `/redteam` -- reads the approved plans and verifies each item exists AND is wired to real data.
2. "Tests pass" had become the definition of done, but 94 backend tests and zero frontend-backend integration tests existed. Fix: convergence criteria now require spec coverage and zero frontend mock data.
3. Todos tracked file creation, not feature completion. Fix: `/todos` now requires every UI todo to have a separate "wire to API" todo, and architecture plans must generate implementation todos.
4. Mock data was invisible to hooks. Fix: `validate-workflow.js` gained `checkFrontendMockData()` that blocks `MOCK_*` constants, `generate*()` fake data functions, and `Math.random()` used as display data in JS/TS.
5. Architecture plans were treated as aspirational, not contractual. Fix: plans approved during analysis must have corresponding implementation todos or be explicitly descoped.

## 4. The Outcome

Six artifacts were changed to close all five gaps. Red team convergence now requires spec coverage verification, increasing round duration but catching coverage gaps. Todo counts increased roughly twofold for frontend features due to the mandatory build/wire split. The `validate-workflow.js` hook blocks frontend files containing mock data patterns. Implementation cycles start with mandatory context re-reading to prevent amnesia across long implementation runs.

The deeper lesson: a quality system that measures only defect density (are there bugs in what exists?) can achieve perfect scores on projects that are fundamentally incomplete. Coverage measurement (does what should exist actually exist?) is a different axis entirely.

## 5. The Spec Connection

**CO §29 (Evidence-Based Completion)** states that completion requires evidence, not assertion. The project asserted completion through passing tests and clean red team rounds, but the evidence supported only a narrow claim: "the code that exists is well-written." The broader claim -- "the project implements what was planned" -- had no evidence at all. §29 requires the practitioner to ask "evidence of what?" before accepting any completion signal.

**CO §3.3 (Safety Blindness)** names the failure mode where the most direct path omits safety considerations. The AI agent completing todos took the most direct path to "file exists" without checking whether the file contained real functionality. §3.3 predicts this: agents optimize for the literal completion criterion, and if the criterion is "file exists," the file will exist -- possibly with mock data.

**CO §49 (Enforcement Reliability)** states that enforcement mechanisms must actually enforce. The `validate-workflow.js` hook detected Python stubs but not JavaScript mock data. The enforcement had a gap in its coverage model. §49 says: if you have a guardrail, verify that it guards the actual attack surface, not just the attack surface you happened to build it for.

**CO §19 (Layer 3 -- Guardrails)** is the architectural layer where deterministic enforcement lives. The five fixes are all Layer 3 changes: adding hook logic, adding phase-gate requirements, adding convergence criteria. This case is a study in Layer 3 incompleteness -- the guardrails existed but had holes that only became visible when a project was sophisticated enough to pass through them.

## 6. Discussion Questions

1. If the project had shipped only backend code (no frontend), would the five process failures have been detected? The mock data was the visible symptom, but failures 1, 3, and 5 (no spec coverage check, todo-tracks-files-not-features, plans-as-aspirational) are backend-agnostic. What would a backend-only project shipping incomplete features look like?

2. The `checkFrontendMockData()` regex targets specific patterns like `generateHourly*` and `MOCK_*`. A developer could write `fetchOccupancy()` that returns hardcoded data without matching any pattern. Is pattern-based detection fundamentally limited for this class of problem, or can the patterns be made comprehensive enough? What would a non-pattern-based approach look like?

3. The post-mortem found that approved architecture plans were treated as aspirational. If the culture had been "plans are contractual," would the data fabric have been implemented as designed, or would it have been descoped during implementation? Which outcome is better -- implementing an over-specified plan or building an ad hoc shortcut that works?
